FestRunner Reference Guide

How to use Jon Gann's film festival template as a reference architecture for building OEFF's Airtable base.

What Is FestRunner?

FestRunner is a 2-base Airtable template built by Jon Gann (DC Shorts Film Festival, Film Festival Alliance founding member) for small film festivals. It's the closest existing structural parallel to what OEFF needs.

Gann migrated from "dozens and dozens of Google sheets" — sound familiar? His core insight: film festival data is inherently relational. Films link to venues, sponsors, reviewers, press. Spreadsheets can't hold those relationships. Airtable can.

Important

FestRunner is a reference architecture, not a starting point. OEFF's 11 cleaned CSVs are the actual import source. Use FestRunner to study how a working film festival base is structured — views, field types, linked records — then apply those patterns to OEFF's schema.

FestRunner's Structure

The system splits into two Airtable bases:

Base 1: FestRunner SUBMISSIONS

Handles the programming pipeline — getting films in, reviewed, and selected.

TablePurpose
FilmsImported from submission engines (FilmFreeway, etc.), tagged by source. Fields: film type, year, imagery, scoring metadata.
ScoresAggregated review data per film.
ReviewsCollected via Airtable forms. Auto-removes films after review thresholds are met.

Base 2: FestRunner EVENTS

The operational base — running the festival once films are selected.

TablePurpose
PeopleAll contacts: filmmakers, press, sponsors, hospitality contacts.
VenuesScreening locations with capacity, AV, contact info.
Shows and EventsScheduled screenings linking films to venues, dates, times.
HospitalityLodging, transportation, guest coordination.
SponsorsSponsorship levels, benefits, contacts.
Print TrafficProgram guides, bookmarks, printed materials tracking.
ShippingPhysical materials logistics.
Marketing OutreachCampaign tracking, social, email, press outreach.
PressMedia contacts, coverage, press kits.

Key Difference

FestRunner splits into 2 bases (Submissions + Events). OEFF puts everything in 1 base. For 739 records, a single base is the right call — it keeps all linked records together and avoids cross-base sync complexity.

Steps: Study FestRunner, Build OEFF

A Explore the Reference
1

Duplicate FestRunner EVENTS into your Airtable

10 min

Go to FestRunner EVENTS on Airtable Universe and click "Use template" to copy it into your workspace.

Don't import any OEFF data into this copy. It's a reference, not a working base. Walk through each table, open each view, click into records to see how fields link together.

2

Study FestRunner's view design

15 min

Gann's key insight: different people see different views of the same data. Press gets a filtered view. Social media gets another. Designers get another. One base, many windows into it.

Notice which fields are hidden in which views. Notice how he uses grouped/sorted views to turn a flat table into something that reads like a schedule or a task list. This is what OEFF needs for Ana vs. the film coordinator vs. the volunteer coordinator.

3

Study how linked records work

10 min

In FestRunner's Shows and Events table, click into a record. Look for fields that link to Venues and People. Click through those links. This is the fundamental thing spreadsheets can't do — and the thing that will make OEFF's data actually usable.

In OEFF's schema: Events links to Venues (via Venue ID), Films (via Film ID). Film Contacts links to Films. Host Intake links to Venues. These are the same pattern FestRunner uses.

B Build the OEFF Base
4

Create a new, empty OEFF base

2 min

Create a fresh base in Airtable called "OEFF 2026". Don't start from FestRunner's template — the schemas don't match. Start empty.

5

Import foundation tables first

15 min

Import in dependency order (as documented in IMPORT-ORDER.md):

  • 01-venues.csv — 100 rows, 9 columns. Primary key: Venue ID
  • 02-films.csv — 12 rows, 14 columns. Primary key: Film ID

These two have no dependencies. Everything else links to them.

6

Import events and link to foundations

20 min

Import 03-events.csv (175 rows, 32 columns). Then create the linked record fields:

  • Convert Venue ID column to a Link to Venues field
  • Convert Film ID column to a Link to Films field

This is the moment it stops being a spreadsheet. When you click a venue in an event record, you'll jump to that venue's full details. This is the linked-records pattern you saw in FestRunner.

7

Create role-based views (steal from FestRunner)

20 min

In the Events table, create these views — modeled on how FestRunner gives different people different windows:

  • "Overview" — all events, grouped by date. Show: Event ID, Date, Venue, Film, Status. Hide the 20+ operational fields.
  • "Film Coordinator" — filtered to 2026 only. Show: Film ID, Film Title, Film Status, Venue, Date, Licensing fields.
  • "Venue Coordinator" — grouped by Venue. Show: Venue, Date, Event Tier, Ticket Price, AV needs, Volunteer Needs.
  • "Dashboard" — filtered to 2026, sorted by date. The at-a-glance view for Ana.

The goal: nobody ever sees 32 columns. Each person sees 5-8 columns that are relevant to their role.

C Import Remaining Tables
8

Import contact and reference tables

15 min
  • 06-sponsors.csv — 23 rows, 36 columns. Link: Event IDs to Events.
  • 11-partners.csv — 150 rows, 6 columns. No links needed.
  • 04-film-contacts.csv — 9 rows, 15 columns. Link: Film ID to Films.
9

Import operational tables

15 min
  • 05-host-intake.csv — 28 rows, 26 columns. Link: Venue ID to Venues.
  • 07-media-assets.csv — 25 rows, 20 columns. Link: Event ID to Events.
  • 08-packet-qa.csv — 23 rows, 18 columns. Link: Event ID to Events.
  • 09-participants.csv — 132 rows, 7 columns.
  • 10-recordings.csv — 62 rows, 5 columns. Link: Event to Events.
D Make It Theirs
10

Record a 3-minute Loom walkthrough

5 min

Screen-record yourself clicking through the base: "Here's where venues live. Here's how you find which film is at which venue. Here's the view you'll use most." This is more valuable than any training doc. Async, rewatchable, zero scheduling overhead.

11

Give the team read-only access

5 min

Share the base with the team as read-only. Ask one question: "Does this data look right?" This starts the adoption clock without the pressure of "you need to learn a new tool." They're just checking data — something they already know how to do.

12

Create a quick reference card

15 min

One page per role. Not a training manual — a cheat sheet. Laminated, taped to desk. Three sections:

  • How to find [the thing they look up most often]
  • How to update [the field they change most often]
  • Who to ask if something looks wrong

Patterns to Steal from FestRunner

Role-specific views, not role-specific tables

FestRunner doesn't create separate tables for press vs. sponsors vs. filmmakers in the People table. It creates views: filtered, sorted, with different visible fields. One table, many lenses. Do the same for OEFF Events.

Forms for external data entry

FestRunner uses Airtable forms for reviewer submissions. OEFF should consider forms for: host venue intake updates, volunteer sign-ups, sponsor benefit confirmations. Forms let people enter data without seeing the full base.

Shared views for external stakeholders

FestRunner shares read-only views with press, designers, etc. OEFF could share a filtered Events view with host venues — they see their screening date, film, and logistics without seeing other venues' data or internal fields.

The "People" consolidation pattern

FestRunner puts everyone in one People table and uses views to filter by role. OEFF chose to split Film Contacts and Participants into separate tables — which is fine for now. But watch for the moment when you need to see "all people connected to Event X." A single People table with a role field might be the Phase 2 move.

What Not to Take from FestRunner

Skip These

The 2-base split. FestRunner separates Submissions from Events. OEFF doesn't need this — all 739 records fit in one base, and a single base keeps all linked records working without cross-base sync hacks.

Skip These

Shipping and Print Traffic tables. OEFF has Packet QA which covers the quality assurance side. Don't add tables you won't use. If print logistics become a pain point later, add the table then.

Skip These

The review/scoring workflow. OEFF's film selection process is different from DC Shorts'. Don't import a review pipeline just because the template has one.

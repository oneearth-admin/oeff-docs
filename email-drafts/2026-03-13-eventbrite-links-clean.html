Quick update — all 25 Eventbrite event pages are cleaned up and working. If you visit any of the event links, you'll land directly on the ticket page. No password prompts, no extra steps.

I thought this was fully sorted last night, but Eventbrite has a few overlapping systems — access codes, event-level passwords, and a separate "Private" publish setting — that each gate the page in different ways. Turned out we had issues in two of those layers, and the second one was harder to spot because it doesn't show up in the places you'd expect.

What was fixed:

  • Removed access codes from all events (leftover from earlier setup — these required visitors to type in a code before seeing tickets)
  • Cleared a privacy setting on 11 newer venue-specific events (Uncommon Ground, IIT, Dominican, etc.) that was blocking the entire page with a password wall

Where things stand now:

  • All 25 events are live but won't show up if someone searches Eventbrite — people can only find them through the direct links we share (from Squarespace, emails, etc.)
  • I've verified every link loads correctly, but please triple-check a few yourselves — extra eyes are worth it here, especially given how many layers Eventbrite has for this stuff

The Squarespace film pages can now link straight to Eventbrite and the ticket flow just works.

Could I have done all of this by clicking through each event one at a time in the Eventbrite dashboard? Sure — and in some ways clicking is more immediately verifiable. But with 25 events it already took a couple passes to get right, and if we scale to 40 or 50 screenings next year, doing this by hand stops being realistic. I built a few small scripts that check every event, fix what needs fixing, and verify the results — hoping we can lean into this kind of tooling more as the festival grows.

For the technically curious:

Eventbrite has three separate systems that can gate an event page, which is part of why this took a couple rounds:

  1. Access codes — codes attached to specific ticket types. Visitors have to enter the code to see those tickets. The API exposes these cleanly, so the script queried each event and deleted any it found. 26 codes removed total.
  2. Event passwords — a password set on the event itself. Shows up as is_locked and password in the API. None of our events had this.
  3. "Private" publish setting — this is the sneaky one. When you create an event through the Eventbrite dashboard and set it to "Private," it flips an internal flag that adds a password gate. But that flag doesn't appear anywhere in the API. If you query the event, every field — is_locked, password, privacy_setting — reads identically to a working event. You can't see it's broken from the data alone.

The fix for #3 turned out to be a toggle: set the event to listed: true via the API (which clears the internal "Private" state), then immediately set it back to listed: false. This resets the event to "unlisted but accessible via direct link" — no password gate, but also not searchable. The original 14 events never had this problem because they were created through the API, which bypasses the dashboard's "Private" mode entirely.

Scripts are in the OEFF data repo if anyone wants to look.
